## 问题诊断与分析

### 问题 1: 用户管理页面空白
**现象**: 访问 `http://localhost:5173/admin/users` 显示空白页面

**原因**: 
- 缺少前端页面组件
- 缺少后端 API 接口
- 路由未配置

### 问题 2: API 请求返回 404
**终端日志**:
```
OPTIONS /api/v1/api/users?page=1&per_page=10 HTTP/1.1" 404
OPTIONS /api/v1/api/users HTTP/1.1" 404
```

**根本原因**: URL 路径重复
- `request.ts` 的 baseURL: `http://127.0.0.1:5000/api/v1`
- `users.ts` 中的路径: `/api/users`
- 实际请求: `http://127.0.0.1:5000/api/v1/api/users` ❌（重复了 `/api`）
- 正确应该是: `http://127.0.0.1:5000/api/v1/users` ✅

### 问题 3: 登录失败 - res.code 为 undefined
**控制台日志**:
```
Login.tsx:21 res.code: undefined
Login.tsx:22 res.data: undefined
```

**原因**: 响应拦截器修改后，返回的数据结构发生变化
- **之前**: 拦截器返回完整的 `AxiosResponse` 对象
- **现在**: 拦截器直接返回 `ApiResponse` 对象（即 `{code, data, message}`）
- Login 组件仍在访问 `response.data`，导致获取到 undefined

// frontend/src/utils/request.ts
request.interceptors.response.use(
  (response: AxiosResponse<ApiResponse>) => {
    const res = response.data;
    if (res.code !== 200 && res.code !== 201) {
      message.error(res.message || '请求失败');
      if (res.code === 401) {
        localStorage.removeItem('token');
        window.location.href = '/login';
      }
      return Promise.reject(new Error(res.message || '请求失败'));
    }
    return res as ApiResponse;  // ✅ 关键修复点
  },

  }
);

## 经验教训

### 1. API 路径设计原则
- baseURL 已经包含版本前缀（`/api/v1`）
- 具体 API 路径不应再包含 `/api`
- 建议统一在 request.ts 中配置 baseURL，各 API 文件只写相对路径

### 2. 响应拦截器设计规范
- 拦截器应该统一返回格式
- 如果返回 `response.data`，所有调用方都要相应调整
- 修改拦截器时要全局搜索所有使用点，确保一致

### 3. TypeScript 类型安全
- 使用类型断言（`as ApiResponse`）确保类型正确
- 接口函数要声明返回类型：`Promise<ApiResponse>`
- 编译时类型检查能帮助发现潜在问题

### 4. 前后端联调最佳实践
- 先测试后端 API（使用 Postman 或测试脚本）
- 确认 API 返回格式后再开发前端
- 使用 console.log 调试时打印完整对象
- 注意区分 `response` 和 `response.data`
