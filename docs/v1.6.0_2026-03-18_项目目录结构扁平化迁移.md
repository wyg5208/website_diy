## 任务目标

将 `company_cms_project/backend` 和 `company_cms_project/frontend` 提升到项目根目录，消除冗余的中间层目录 `company_cms_project/`，使项目结构更加扁平和清晰。

## 问题诊断与分析

### 问题 1: 目录层级冗余
**现象**: backend 和 frontend 嵌套在 company_cms_project/ 下，增加了路径复杂度  
**原因**: 早期项目结构设计，随着项目发展需要扁平化

### 问题 2: SQLite 数据库路径解析问题
**现象**: 迁移后后端启动报错 `sqlite3.OperationalError: unable to open database file`  
**原因**: Flask-SQLAlchemy 将相对 SQLite 路径自动拼接到 Flask 的 `instance_path`（backend/instance/），而非期望的 backend/ 目录  
**解决**: 在 config.py 中使用 basedir 构建绝对路径

## 技术实现细节

### config.py 数据库路径修复

核心修改：使用 `basedir` 构建绝对路径，避免 Flask-SQLAlchemy 的 `instance_path` 拼接问题。

```python
import os
from datetime import timedelta
from dotenv import load_dotenv

# 加载.env文件
load_dotenv()

# 获取当前文件所在目录的绝对路径
basedir = os.path.abspath(os.path.dirname(__file__))

class Config:
    """基础配置类"""
    # Flask基础配置
    SECRET_KEY = os.getenv('SECRET_KEY', 'fallback-secret-key')
    DEBUG = os.getenv('DEBUG', 'False').lower() == 'true'
    
    # 数据库配置 - 使用绝对路径避免 Flask instance_path 解析问题
    SQLALCHEMY_DATABASE_URI = os.getenv('DATABASE_URL', f'sqlite:///{os.path.join(basedir, "data", "cms.db")}')
    SQLALCHEMY_TRACK_MODIFICATIONS = False
    SQLALCHEMY_ECHO = DEBUG  # 开发环境下打印SQL语句
```

**关键点**:
- `basedir = os.path.abspath(os.path.dirname(__file__))` 获取 config.py 所在目录（backend/）
- `os.path.join(basedir, "data", "cms.db")` 构建绝对路径，确保指向 `backend/data/cms.db`

## 经验教训

### 1. Flask-SQLAlchemy 路径解析陷阱
Flask-SQLAlchemy 会将相对 SQLite 路径自动拼接到 `instance_path`，建议始终使用基于 basedir 的绝对路径。

### 2. 渐进式迁移策略
采用"先复制、再修改、后验证"的策略，保留原始目录作为回滚方案，有效降低了迁移风险。
